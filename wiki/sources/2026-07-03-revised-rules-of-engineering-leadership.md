## Summary
Will Larson(전 Uber, 현 Imprint 엔지니어링 리더)이 2014~2020년 하이퍼그로스 환경에서의
경험과, 최근 AI 코딩 도구가 가져온 작업 속도 변화를 바탕으로 엔지니어링 리더십
원칙을 다섯 가지로 재정리한 글. 각 원칙마다 Imprint에서 실제로 진행한 프로젝트
(배포 자동화, 모노레포 전환, 설정 시스템 통합, 프론트엔드 타입화, 이슈 트리아지
자동화 등)를 근거로 든다. 핵심 논지는 "기술적 실행 속도는 극적으로 빨라졌지만,
조직의 병목(정렬 부족, 우선순위 불명확, 부실한 아키텍처)은 예전과 다르지 않다"는 것.

## Key claims
- **개인 주도 마이그레이션**: 복잡하고 큰 변경도 주도하는 개인/소수팀이 95%를
  소유하고 예전 대비 10%의 시간에 끝낼 수 있게 되었다. 마이그레이션 비용이 낮아질수록
  개인 판단력의 질이 갖는 보상/페널티는 오히려 커진다 — 작은 결함도 동료의 코드
  멘탈 모델을 깨뜨리기 때문. 자세한 내용: [[concepts/engineering-leadership]].
- **작동하는 코드에는 실제 비용이 든다**: 1차 초안 코드는 거의 공짜가 됐지만,
  "작동하는" 코드의 비용은 개발 하네스(테스트, CI/CD, 검증 환경, 변경 미리보기 등)에
  달려 있고 결코 공짜가 아니다. 2년 전 엔지니어링 속도를 높이는 데 가장 가치 있었던
  요소가 지금도 여전히 가장 가치 있다. 자세한 내용: [[concepts/engineering-leadership]].
- **프로세스의 기본 케이스를 에이전트용으로 자동화**: 대부분 프로세스의 기본
  케이스는 적절한 하네스·통제·도메인 맥락·설계자의 판단력이 갖춰지면 완전 자동화할
  수 있다. 예: 사람이 하는 1차 코드 리뷰보다 좋은 하네스의 코드 리뷰가 더 빠르고
  효과적이다. 고위험 영역만 사람이 처리하면 된다. 스프린트 등 계획 프로세스는
  너무 낮은 고도(altitude)에서 작동하고 있다고 지적한다.
  자세한 내용: [[concepts/engineering-leadership]].
- **도메인 맥락을 가진 지속적(durable) 팀이 더욱 중요해졌다**: 개인 생산성이
  올라가도, 좋은 판단에는 여전히 도메인 맥락이 필요하며 이는 지속적으로 한 영역에서
  일해온 팀에서 축적된다. "소수의 천재 엔지니어가 모든 걸 처리하는" AI 시대 비전에
  대해 저자는 명시적으로 동의하지 않는다 — 고judgment 개인도 결국 도메인 맥락
  부족에 막힌다고 본다. 자세한 내용: [[concepts/engineering-leadership]].
- **빠르고 좋은 durable 의사결정이 AI 이득을 실현하는 전제조건**: 실행 속도가
  빨라진 만큼, 그 이득을 누리려면 조직이 빠르고 되돌리지 않는 결정을 내려야 한다.
  이 때문에 평균적인 CTO 역할이 1년 전보다 훨씬 기술적이고 덜 관료적으로 변했다고
  주장한다. 자세한 내용: [[concepts/engineering-leadership]].

## Notable specifics
- Imprint의 배포 빈도: 1년 전 수동 배포 주 ~6회 → 현재 주 200~400회. 엔지니어링
  인원은 2배 늘었지만 그걸 감안해도 연간 20~30배 증가.
- 1월 1일 기준 팀의 ~25%가 매일 Claude Code/Cursor 사용 → 2월 말 100%. 별도의
  하향식 지시 없이, 도구를 좋게 만들고 비채택자와 대화하는 방식만으로 달성.
- "지금은 거의 모든 PR이 최소 1차는 하네스가 작성한다."
- 설정 메커니즘 다수 → 2가지로 통합: 예전이었다면 수년 걸렸을 프로젝트를 한 분기
  안에 완료(엔지니어들이 순차적으로 레퍼런스 아키텍처를 따라간 방식).
- 멀티레포 프론트엔드 → 모노레포 전환: 한 명의 프론트엔드 엔지니어가 95% 주도,
  약 한 달 소요.
- 프론트엔드 코드 전체 정적 타입화: 한 명의 엔지니어가 몇 주에 걸쳐 완료.
- npm → pnpm 마이그레이션: 한 명이 하루 몇 시간씩 며칠 만에 완료.
- 고객 이슈 트리아지를 하네스가 담당(팀·오픈 티켓 인지, 데이터 웨어하우스 제한
  접근으로 이슈 영향도 산정) — 엣지케이스만 사람이 트리아지.
- "1차 코드 리뷰는 변경을 구현한 것과 같은 하네스가, 그 변경을 작성할 때 쓴 맥락을
  비운 상태로 수행해 사람은 더 높은 가치의 피드백에 집중한다."
- Linear로 전환(Jira에서), 더 나은 MCP·Slack 연동으로 에이전트 우선 워크플로를
  지원 — Linear 이슈를 자동으로 가져와 해결을 시도하는 내부 하네스를 알파 테스트 중.
- 경고: "다른 팀 엔지니어에게 설계 문서·PR을 '담 너머로' 던지는 시도는 실제로
  성사된 적이 없다. 슬롭(slop) PR과 설계 문서는 만들기는 싸지만 적극적으로 해롭다
  — 정리·수리 비용이 들 뿐 아니라 그 맥락이 LLM을 오염시켜 처음부터 다시 하는 것보다
  나쁜 결과를 낳는다."
- 매니저가 직접 코드를 기여해 성공한 사례는, 매니저가 변경 사항을 직접 검증하고
  배포 후 대시보드를 확인하며 문제를 직접 해결하는 경우에 한정된다고 관찰됨.
