# Google Code Review

**Raw file:** `raw/202607/20260703-google-code-review.md` · **Type:** 웹 기술 문서 (연작 시리즈)
**Author/origin:** Google Engineering Practices 공식 문서 · **Published:** 확인 불가 (raw 파일에 날짜 없음)

## Summary
Google 엔지니어링 실천 가이드 중 "리뷰어" 파트에 속한 6개 페이지 시리즈
(`eng-practices/review/reviewer/`)를 함께 조회한 결과다. 코드 리뷰어가 지켜야 할
표준, 리뷰 시 확인할 항목, 변경사항(CL)을 훑어보는 순서, 리뷰 속도, 코멘트 작성법,
저자의 반발(pushback)에 대응하는 법을 각각 다룬다. raw 파일 자체에는 URL 목록만
있었고 본문은 각 URL을 직접 조회해 확보했다.

## Key claims

### 리뷰 표준 (standard.html)
- 리뷰어는 CL이 완벽하지 않더라도 "전체 코드 상태(code health)를 확실히 개선한다"고
  판단되면 승인해야 한다. 완벽한 코드란 없고, 더 나은 코드가 있을 뿐이다.
  [[concepts/code-review]]
- 사소한 다듬기(polish) 문제로 승인을 며칠~몇 주씩 지연시키지 말고, 필수가 아닌
  제안은 "Nit:" 라벨을 붙인다.
- 기술적 사실과 데이터가 개인 의견보다 우선하며, 스타일 가이드는 스타일에 관한
  절대적 권위를 가진다.
- 여러 유효한 접근법이 있고 저자가 자신의 선택이 타당함을 보일 수 있다면 저자의
  선호를 받아들인다.
- 코드 상태를 명백히 악화시키는 변경은 진짜 비상 상황이 아닌 한 승인해서는 안 된다.
- 의견 충돌 시에는 합의를 모색하고, 필요하면 대면 논의를, 그래도 안 되면 상급자
  에스컬레이션을 통해 해결하며 무기한 보류하지 않는다.

### 리뷰 시 확인할 항목 (looking-for.html)
- 설계(Design): 전체 아키텍처가 타당한지, 이 코드베이스에 속해야 하는지(혹은
  라이브러리로 가야 하는지), 기존 시스템과 잘 통합되는지, 지금이 이 기능을 추가할
  적절한 시점인지 평가한다.
- 기능(Functionality): 변경이 개발자의 의도를 달성하는지, 엣지 케이스와 동시성
  문제를 고려하는지 본다. "사용자에게 영향을 주는 변경(예: UI 변경)일 때 동작 확인이
  가장 중요하다."
- 복잡도(Complexity): 라인·함수·클래스 수준에서 불필요한 복잡도를 지적한다.
  복잡도란 "코드 독자가 빠르게 이해할 수 없는 상태"를 뜻하며, 특히 현재 필요 없는
  미래 요구를 미리 대비하는 과잉 설계(over-engineering)를 경계한다.
- 테스트(Testing): 프로덕션 코드와 함께 적절한 단위/통합/E2E 테스트를 요구하고,
  테스트가 실제로 코드가 깨질 때 실패하는지, 거짓 양성(false positive)은 없는지
  확인한다.
- 네이밍과 주석: 이름은 충분히 설명적이되 과도하게 길지 않아야 하며, 주석은 코드가
  "무엇을" 하는지가 아니라 "왜" 존재하는지를 설명해야 한다.
- 스타일과 일관성: 스타일 가이드를 지키게 하되, 필수가 아닌 선호는 "Nit:"으로
  표시한다. 기존 코드가 가이드와 다르더라도 스타일 가이드가 우선한다.
- 문서화: 빌드/테스트/사용법에 영향을 주는 변경이면 관련 문서를 함께 갱신해야 한다.
- 리뷰어는 배정된 모든 라인을 검토해야 하고, 잘한 부분은 칭찬해야 하며, 보안·
  접근성처럼 전문성이 필요한 영역은 해당 분야 리뷰어가 커버해야 한다.

### CL을 훑어보는 순서 (navigate.html)
- 3단계 접근: (1) CL 설명을 읽고 이 변경이 애초에 존재해야 하는지 큰 그림에서
  판단, (2) 논리적 변경이 가장 많은 주요 파일부터 검토해 근본적 설계 문제를
  찾아내기, (3) 나머지 파일을 논리적 순서로 검토.
- 근본적인 문제가 있다면 세부 구현을 검토하느라 시간을 낭비하기 전에 일찍 반려한다.
- 큰 설계 피드백은 최대한 빨리 보낸다. 개발자가 이미 이 CL을 기반으로 다음 작업을
  시작했을 수도 있고, 재설계는 시간이 더 걸리므로 마감을 맞추려면 빨리 알아야 한다.
- 구현 코드보다 테스트를 먼저 읽어 의도된 동작을 파악하는 것도 방법이다.
- 반려하더라도 대안을 제시하고 개발자의 노력을 존중하는 태도를 유지한다.
- 여러 CL이 반복적으로 원치 않는 방향이라면, 사후 반려보다 사전에 팀 커뮤니케이션
  프로세스를 재검토하는 편이 낫다.

### 리뷰 속도 (speed.html)
- Google은 개인 개발자의 속도보다 팀 전체의 속도(velocity)를 우선시한다.
- 느린 리뷰의 3가지 문제: 기능/버그 수정이 며칠~몇 주씩 지연됨, 리뷰어가 뜸하게
  응답하면서 큰 수정을 요구하면 개발자가 불만을 갖게 됨, 리뷰가 밀리면 저품질
  코드를 승인하라는 압박이 커지고 개발자가 정리(cleanup)·리팩터링 작업을 피하게 됨.
- "코드 리뷰 프로세스에 대한 불만 대부분은 실제로는 프로세스를 덜 까다롭게 해서가
  아니라 더 빠르게 만듦으로써 해결된다."
- 응답 시간 기준: 집중 작업 중이 아니라면 즉시 응답하고, 리뷰 요청에 응답하는 데
  걸리는 최대 시간은 "영업일 기준 1일"이다. 일반적인 CL은 하루 안에 여러 차례
  리뷰 라운드를 거쳐야 한다.
- 핵심은 전체 리뷰 사이클을 빠르게 끝내는 것보다 "빠른 응답" 자체다.
- 집중해서 코딩 중인 개발자를 인터럽트하지 말고 자연스러운 휴식 지점을 기다린다.
- 지나치게 큰 CL은 더 작고 순차적인 리뷰로 쪼개도록 요청한다.
- 시간대가 다른 개발자를 배려해 상대방의 근무일이 끝나기 전에 리뷰를 마친다.
- 사소한 코멘트가 남아있어도 "LGTM with comments"로 승인해 하루 종일 지연되는
  것을 막는다.

### 코멘트 작성법 (comments.html)
- "친절하게(Be kind)" 대하고, 코멘트는 개발자가 아니라 코드에 대해서만 작성한다.
- 피드백에는 이유를 함께 제시해 개발자가 "왜"를 이해하고 학습하도록 돕는다.
- 문제를 고치는 책임은 리뷰어가 아니라 개발자에게 있다. 문제 지적과 직접적인 해결책
  제시 사이의 균형을 맞추고, 개발자가 스스로 결정하게 두면 더 나은 해법과 더 빠른
  학습으로 이어지는 경우가 많다.
- 약점뿐 아니라 잘한 점(깔끔한 알고리즘, 좋은 테스트 커버리지, 참신한 접근 등)에도
  코멘트를 남긴다.
- 코멘트 심각도를 라벨로 표시: "Nit:"(사소한 문제), "Optional/Consider:"(필수가
  아닌 제안), "FYI:"(정보 제공용) — 모든 코멘트가 필수 요구사항으로 오해되는 것을
  방지한다.
- "코드 리뷰 도구에만 적힌 설명은 미래의 코드 독자에게 도움이 되지 않는다."
  코드가 복잡해서 설명이 필요하다면, 리뷰 도구에 설명을 남기는 대신 코드를 다시
  쓰거나 인라인 주석을 추가하도록 요청한다.

### 저자의 반발(pushback)에 대응하기 (pushback.html)
- 개발자가 리뷰 의견에 반박하면 먼저 그 말이 맞는지 판단한다. "종종 그들이 리뷰어
  보다 코드에 더 가까이 있으므로, 특정 측면에 대해 실제로 더 나은 통찰을 가지고
  있을 수 있다." 근거가 타당하면 인정하고 넘어간다.
- 그래도 그 변경이 코드 상태를 개선한다고 판단되면 계속 주장한다. "코드 상태
  개선은 작은 단계로 일어나는 경향이 있다"는 것을 근거로, 여러 차례 설명을
  반복하더라도 전문성 있는 태도를 유지한다.
- 엄격한 리뷰가 개발자를 화나게 할 것이라는 우려에 대해, 실제로는 정중한 태도로
  코멘트를 남기면 개발자는 화내지 않는 경우가 많고, 오히려 나중에는 개선에
  고마워하는 경우가 많다고 말한다.
- 개발자가 "나중 CL에서 정리하겠다"고 요청하는 경우가 흔한데, 이를 강하게 만류
  한다. "원래 CL을 작성한 후 시간이 지날수록 그 정리 작업이 실제로 이루어질
  가능성은 낮아진다." 정리는 즉시 하거나 아예 안 하는 것으로 취급해야 한다.
- 더 엄격한 리뷰로 전환할 때는 초기에 불만이 나올 수 있지만, 리뷰 속도를 개선하면
  이런 불만이 줄어들고 결국 개발자들도 엄격한 기준의 가치를 인정하게 된다.
- 그래도 해결되지 않는 갈등은 "The Standard of Code Review" 문서(standard.html)의
  갈등 해결 프레임워크를 참고한다.

## Notable specifics
- 리뷰 요청 응답 시간 상한: 영업일 기준 1일.
- 시리즈 URL 6개는 모두 `google.github.io/eng-practices/review/reviewer/` 하위
  경로이며, 서로 다른 하위주제(standard, looking-for, navigate, speed, comments,
  pushback)를 다룬다.
